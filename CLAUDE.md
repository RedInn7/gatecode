
# # Role: 高级算法测试审计员 (Data Quality Auditor)

## 🏗️ 核心任务 (Core Mission)

作为 Gatecode 的质量守门员，你的任务是为每一道题目构建**工程级、具有高辨析度**的测试数据集。你必须执行【自我审计】流程，确保输出的 JSON 数据不仅格式正确，且在**逻辑深度、分布多样性、极端边界**上无懈可击，能够有效区分“最优解”、“暴力解”与“错误解”。也不一定非得用golang，用你觉得最合适的技术栈就行

---

## 🔍 第一阶段：审计与修正逻辑 (Audit & Refinement)

在输出最终 JSON 前，必须通过以下三层审计过滤。**若不满足，必须重构数据。**

### A. 逻辑一致性 (Logical Integrity)

* **约束校验**：严禁违反题目性质。你需要使用agent进行读题确保满足题意
* *BST题目*：数组序列必须严格满足二叉搜索树中序遍历规律。
* *DAG题目*：边集构造必须进行拓扑检测，严禁出现隐性环。
* *合法性*：所有 `k`、`index` 参数必须处于 `[min, max]` 合法区间内。



### B. 分布多样性 (Distribution Diversity) —— **拒绝全0/全1/单一化**

* **密度控制**：单一样本中，若 80% 以上元素相同（如全是 0/1/相同字符），该样本无效，必须重新生成。除非题目有特殊要求。
* **值域覆盖**：强制覆盖题目值域的 `Positive`, `Negative`, `Zero`, `MAX_INT`, `MIN_INT`。除非题目有特殊要求。
* **拓扑结构**：
* *树/图*：必须包含：满二叉树、完全左斜树、完全右斜树、随机稀疏图、完全图。除非题目有特殊要求。
* *字符串*：混合大小写、数字、特殊符号及长短交替分布。除非题目有特殊要求。



### C. 极端边界注入 (Edge Case Injection)

**每 20 组测试点中，必须包含以下“杀手级”席位：**

* **零值/极小值**：`n=0`, `n=1`, `""` (空串), `[]` (空阵), 只有根节点的树。除非题目有特殊要求。
* **极端形态**：单调递增序列、单调递减序列、全等元素序列。除非题目有特殊要求。
* **溢出临界**：数值紧贴  或 ，以及大数求和溢出测试。除非题目有特殊要求。

---

## ⚙️ 第二阶段：判题执行协议 (Judge Execution Protocol)

### 1. 结果正确性判定 (Correctness)

* **Standard Diff**: 默认模式。严格逐行比对，忽略行末空格及文件末尾换行。
* **Special Judge (SPJ)**: 针对结果不唯一（如浮点精度、路径规划、任意解返回）。
* *Action*: 若 `problems.is_spj = true`，必须调用指定校验脚本进行逻辑验证。



### 2. 资源限制与状态映射 (Status Mapping)

| 状态 (Status) | 触发条件 | 处理逻辑 |
| --- | --- | --- |
| **AC (Accepted)** | 结果完全正确 | 记录通过。 |
| **WA (Wrong Answer)** | 结果比对失败 | 指出首个错误位置。 |
| **TLE (Time Limit)** | 运行超时 | C++ 1s / Java-Python 2s。 |
| **MLE (Memory Limit)** | 内存超限 | 默认 256MB（包含 JVM 开销）。 |
| **RE (Runtime Error)** | 段错误、数组越界、异常 | 捕获系统信号并返回错误码。 |
| **OLE (Output Limit)** | 输出文件 > 32MB | 强制终止，防止磁盘写满。 |

### 3. 安全隔离与审计 (Security & Isolation)

* **系统调用限制**：禁止 `fork`, `execve`, `socket`, `mount` 等。
* **网络策略**：容器强制执行 `network: none`。
* **文件系统**：`Read-only` 挂载系统目录，仅允许 `Read/Write` 指定 `/tmp` 与测试目录。

---

## 🚀 第三阶段：自动化验证流 (Verification Flow)

1. **并行验证**：在生成 Output 后，必须并行运行“标准程序 (Standard Solution)”进行交叉验证。
2. **不一致处理**：若 Output 与标准程序不符，优先回溯检查**输入数据 (Input)** 是否满足题目约束，而非盲目修改 Output。
3. **自修复循环**：发现 Bug 后，必须完成“修复 -> 重新运行 -> 润色整合”的全过程。
4. **多语言验证**:你还需要运行java, cpp, 等我们目前所拥有的所有语言的标准程序，确保后端判题机能处理所有的语言。

Gatecode 判题引擎 (Judge Engine) 需求与实现目标文档
1. 核心定义 (Core Definition)
判题引擎是一个高度隔离的计算单元，负责接收用户代码及其对应的测试数据，在受限环境下运行并对比输出结果，最终返回结构化的评测报告。

2. 核心功能目标 (Functional Goals)
A. 多模式运行能力
ACM 模式支持 (#6)：支持标准输入流（stdin）注入和标准输出流（stdout）捕获。能够处理单文件多组测试用例。

LeetCode 模式支持：支持核心函数式判题。引擎需具备“代码拼接”能力，将用户提交的函数片段与预设的 Driver（主程序）合并编译。

多语言兼容性：统一的接口规范，支持编译型语言（C++, Java, Go）和解释型语言（Python, Node.js）。

B. 资源监控与限制
时间精度控制：必须能够精确到毫秒（ms）级的运行时间统计，并支持硬性超时截断（TLE）。

空间精确控制：必须监控峰值内存占用（MB/KB），严禁进程使用 Swap，达到阈值立即终止（MLE）。

输出大小限制：限制程序向 stdout 或 stderr 写入的数据总量（如 32MB），防止磁盘撑爆。

C. 判题结果分类 (Verdict)
引擎必须返回以下标准状态之一：

Accepted (AC)：答案完全正确。

Wrong Answer (WA)：输出与标准答案不符。

Time Limit Exceeded (TLE)：运行超过规定时间。

Memory Limit Exceeded (MLE)：内存超过规定限制。

Runtime Error (RE)：程序异常退出（段错误、除零等）。

Compile Error (CE)：代码无法通过编译。

System Error (SE)：沙箱环境异常（如配置错误）。

3. 安全防御目标 (Security Requirements)
这是 Gatecode 的生命线，必须确保：

网络阻断：判题过程中进程必须完全断网，防止代码将题目数据外泄或发起 DDoS 攻击。

文件系统隔离：代码仅能访问必要的运行库，禁止读写宿主机敏感目录（如 /etc, /root）。

系统调用拦截：通过黑白名单机制，禁止执行危险系统调用（如 mount, kill, fork 炸弹防御）。

无状态化：每次判题结束后，环境必须物理清空，确保不同用户的代码不会互相干扰。

4. 商业化与性能目标 (Commercial & Performance)
A. 优先级调度 (VIP Priority)
需求：支持队列优先级。

目标：当 VIP 用户（来自 #26 Stripe 接入）提交代码时，判题请求应优先于普通用户进入计算单元，缩短排队时间。

B. 判题效率
并发处理：支持横向扩展。当用户量激增时，可以通过增加判题节点无缝扩容。

冷启动优化：预热常用语言环境镜像，确保单次判题的系统开销（Overhead）降至最低。

每隔5分钟需要输出进度一次，碰到没通过的题目，首先要确保测试input在题目要求的范围内

# 一、快：反馈循环决定用户粘性

## 🎯 留存痛点

* 提交代码后转圈超过 **5 秒**，用户会开始：

  * 切换标签页
  * 打开微信
  * 刷短视频
* 注意力一旦转移，任务中断概率急剧上升
* 没有“进度感” → 焦虑 → 失去信任

---

## 🔥 1. 毫秒级反馈与流式状态推送

### ✅ 实时流式反馈（Streaming Updates）

判题机不能等所有测试点跑完才返回结果。

**实现策略：**

* 使用 `WebSocket` 或长轮询
* 每跑完一个测试点，立即推送状态

示例：

```
Compiling...
Running Test 1/20...
Running Test 2/20...
[2/20] Passed
```

这种“进度感”本身就是心理安抚机制。

---

### ✅ 静态预检查（Static Pre-check）

在进入沙箱前，先做：

* 语法快速检测
* 编译前静态检查

若存在低级语法错误：

* 100ms 内返回 CE
* 不占用沙箱资源
* 不进入队列

---

### ✅ 编译优化与镜像预热

问题：C++ 和 Java 编译时间过长。

解决方案：

* 判题镜像预热
* 挂载缓存目录
* 预编译常用库：

  * `bits/stdc++.h`
  * Java 常用工具类
* 构建长期运行的 Worker 容器

目标：

> 将编译时间压缩至 **1 秒以内**

---

### ✅ VIP 优先调度机制

在 Redis 队列中实现 **Priority Queue**：

* 普通用户：默认权重
* VIP 用户：最高权重

高峰期（晚 8-11 点）保障：

* 排队时间 `< 500ms`
* 总反馈时间 `< 2s`

商业价值：
**“快”是最直观的会员感知差异。**

# 二、准：信任是留存的基石

## 🎯 留存痛点

* 本地 AC，线上 WA
* 本地运行正常，线上 RE
* 平台不给解释

一旦用户产生：

> “这平台判题不准”

他将永远不会回来。

---

## 🎯 1. 判题环境镜像一致性

### ✅ 发布标准环境清单

在前端显著位置展示：

```
C++: g++ 11 -O2
Java: OpenJDK 17
Python: 3.10
```

必须保证：

* 本地可复现
* 环境版本稳定
* 不随意升级

---

### ✅ Constraints 强制对齐

如果题目写：

```
N ≤ 10^5
```

那么判题机必须：

* 分配足够堆内存
* 支持该规模数组
* 不因环境问题导致 MLE

---

### ✅ Java 内存策略优化

JVM 启动本身占用较大。

策略：

* 自动增加 256MB 堆外余量
* 避免语言差异导致误判 MLE

---

### ✅ 多语言时限对等（2× Rule）

默认规则：

```
Java TL = 2 × C++ TL
```

避免 Java 因常数项大而无故 TLE。

---

## 🎯 2. 错误信息结构化输出

### ❌ 不要：

```
Runtime Error
Wrong Answer
```

### ✅ 应提供：

#### WA：

* 显示前 100 个字符差异
* 对比：

  * Expected Output
  * Your Output

#### RE：

* 捕获 stderr
* 展示核心异常：

```
Exception in thread "main"
java.lang.NullPointerException
```

* 隐藏系统路径
* 保留核心错误类型

---

### ✅ 商业化差异

VIP 可查看：

* 导致错误的输入样例
* 详细调试信息
* 对应题解入口

这对于：

* 题解系统
* 精选题单

具有极高商业转化价值。

# 三、稳：一致性决定品牌价值

## 🎯 留存痛点

> “刚才 AC，现在 TLE 了？”

这是 OJ 平台的自杀行为。

---

## 🎯 1. 资源刚性隔离

必须实现：

```
docker run \
  --cpus=".5" \
  --memory="512m" \
  --pids-limit=50
```

不能使用软限制。

---

### ✅ cgroups 强隔离

* 限制 CPU 核心占用
* 限制内存
* 限制进程数

确保：

> 一个死循环不会拖慢整台服务器

---

## 🎯 2. 系统调用黑名单（Seccomp）

必须禁用：

* `fork`（防止 fork 炸弹）
* 网络相关 syscall
* `mount`
* 文件系统越权调用

---

## 🎯 3. 双层超时机制

### 第一层：沙箱内 Alarm

* 控制代码逻辑超时

### 第二层：沙箱外 Killer 进程

* 超过 5s 强制 `SIGKILL`
* 防止僵尸进程

---

## 🎯 4. 确定性判题（Deterministic Judging）

目标：

> 同一份代码，在任何时间提交
> 运行时间波动控制在 ±5% 内

实现方式：

* 固定 CPU 份额
* 禁止宿主机干扰
* 统一运行环境

---

## 🎯 5. 容器生命周期优化

❌ 不要频繁启停容器

### ✅ 推荐策略：

* 一次启动
* 多次输入
* 批量执行
* 统一销毁

优点：

* 减少冷启动开销
* 提高 QPS
* 降低系统抖动
